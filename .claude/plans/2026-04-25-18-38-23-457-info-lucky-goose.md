# 审视与优化计划（排除上下文压缩）

## 背景

参考 `PROJECT_TREE_AND_DATA_FLOW.md`，dcp_optimizer.py 已集成完善的子系统（`MemoryManager`、`EventBus`、`YAMLStructuredCompressor`），但 2026-04-25 的运行日志暴露了一个关键假阳性：WNS=1393.000 ns 被接受并触发提前终止，掩盖了真实进度（从 -0.978 改进到 -0.496）。

## 问题 1: WNS 假阳性链（关键）

**根因链**（参考 `PROJECT_TREE_AND_DATA_FLOW.md` 第 2.4 节）：

1. **`vivado_mcp_server.py:get_wns` (行 1486)**：`wns_value = output.strip().split('\n')[-1].strip()` 取 pexpect 缓冲区的最后一行。如果前一个 `run_tcl_command` 的输出（如 `set_property IS_SOFT 0 [get_pblocks pblock_opt]`）泄漏到缓冲区，最后一行不是 WNS 数值。

2. **`run_tcl_command` pexpect 缓冲区泄漏**：`VIVADO_PROMPT = r"\r?\nVivado% "` 可能过早匹配，剩余输出泄漏到下一个命令的 `proc.before`。

3. **`MemoryManager.add_tool_result()` (manager.py:281)**：`if wns is not None and wns > self._best_wns` — **没有任何 WNS 合理性验证**。任何浮点数 > best_wns 都被接受。

4. **`dcp_optimizer.py:call_tool()` (行 1456-1514)**：WNS 通过三条路径被捕获，每条都直接调用 `float()`，**没有边界检查**。

5. **`get_completion()` (行 1929-1930)**：`current_best_wns >= self.WNS_TARGET_THRESHOLD (0.0)` 宣布收敛，没有检查该值是否合理。

**这是日志中 WNS=1393.000 的直接原因链**。此外，`dcp_optimizer.py:call_tool()` 和 `MemoryManager.add_tool_result()` 之间通过 `_compat.add_tool_result()`（dcp_optimizer.py:1523）存在 WNS 更新的**双重路径**。

## 问题 2: 无限制的内部工具循环（高）

`dcp_optimizer.py:get_completion()` (行 1774)：
```python
while True:
    # ... LLM 调用
    # ... 处理工具调用
    continue  # 行 1905 — 无限循环
```

内部工具轮次没有上限。第 1 次迭代使用了 24 次 API 调用，消耗了约 48 秒的持续 API 使用，然后**触发了 DeepSeek V4 Flash 上的速率限制**。每次轮次都会向上下文添加约 1K 个令牌（~500 个助理令牌 + ~500 个工具结果令牌）。

## 问题 3: 模型选择偏差（中）

来自 `PROJECT_TREE_AND_DATA_FLOW.md` 第 2.5 节：
- 当分数接近时，默认值为 planner（pro 模型）
- 决策阈值：`planner_score > worker_score + 1`（偏差 2 分）
- `TOOL_MODEL_MAPPING` 不完整 —— `search_sites`、`get_tile_info`、`vivado_run_tcl` 未映射，得分为 0

这导致 Flash 模型（便宜 14 倍）的使用不足。日志显示在迭代 5-6（flash→pro→flash→pro）期间模型抖动。

## 问题 4: 速率限制失败恢复（中）

`dcp_optimizer.py` 行 1793-1820：API 重试等待时间为 `2^(retry)`（2 秒、4 秒、8 秒），然后传播异常。

OpenAI SDK 在应用重试之前还会进行 3 次内部重试（间隔约 0.4 秒、约 0.8 秒、约 0.4 秒）。**总共 9 次重试尝试**在失败前浪费了约 15 秒。

当速率限制最终失败时，异常处理程序（行 2134）添加了一条用户消息，但**没有切换模型**。模型切换仅在下一个迭代中的 `_select_model()` 中发生，该函数不知道速率限制。切换到 Mimo 2.5 Pro 增加了约 3-15 倍的成本。

## 问题 5: 冗余的 global_no_improvement 递增（低）

`global_no_improvement` 在两个地方递增，参考点不同：
- `get_completion()`（行 1920）：比较最新工具调用 WNS 与 `iteration_start_wns`
- `_on_iteration_end()`（通过 `wns_improved`）：比较迭代前后的 `best_wns`

`get_completion()` 中的递增是不必要的并且可能导致比预期更早地达到 `GLOBAL_NO_IMPROVEMENT_LIMIT` 阈值。

## 计划变更

### 变更 1：添加 WNS 合理性验证

**新的辅助方法** `_is_valid_wns(wns: float) -> bool` 添加在 `dcp_optimizer.py` 中：

```python
def _is_valid_wns(self, wns: float) -> bool:
    if wns is None:
        return False
    # WNS 不应超过时钟周期的 10 倍
    if self.clock_period and abs(wns) > self.clock_period * 10:
        return False
    # 极端负值通常是解析错误
    if wns < -999:
        return False
    # 从显著负值跳转到恰好 0.0 而无优化步骤则是可疑的（警告但不拒绝）
    if wns == 0.0 and self.best_wns < -0.1:
        logger.warning(...)
    return True
```

**在 `dcp_optimizer.py:call_tool()` 中的 3 个 WNS 跟踪点应用**：
- 行 1459（auto-eval 后期物理优化）：在 `current_wns > self.best_wns` 之前添加验证
- 行 1476（`report_timing_summary` 路径）：在更新 `self.best_wns` 之前添加验证
- 行 1498（`get_wns` 路径）：**关键** —— 在更新 `self.best_wns` 之前添加验证（这将捕获 1393.000）

**在 `dcp_optimizer.py:get_completion()` 完成检查（行 1929-1930）应用**：在因 `wns_target_met` 返回 `is_done=True` 之前，验证 `current_best_wns` 是合理的（不大于时钟周期的 10 倍）。

### 变更 2：修复 get_wns Vivado MCP Pexpect 缓冲区泄漏

**文件**：`VivadoMCP/vivado_mcp_server.py` (行 1478-1487)

**之前**（有缺陷）：
```python
tcl_cmd = "set wns_path [get_timing_paths -max_paths 1 -slack_lesser_than 999]; if {[llength $wns_path] > 0} {get_property SLACK $wns_path} else {puts 0.0}"
output = run_tcl_command(tcl_cmd, timeout=timeout)
wns_value = output.strip().split('\n')[-1].strip()  # 获取最后一行 — 脆弱！
```

**之后**（健壮）：
- 将 Tcl 命令简化为：`puts [get_property WNS [current_design]]` — 直接询问 Vivado 当前的 WNS。这是单行输出，没有路径查询开销。
- 用多策略解析替换脆弱的最后一行解析：
  1. 尝试从删除空白后的输出中提取浮点数 `[-]digits.digits`
  2. 使用 `re.search(r'-?\d+\.\d+', output)` 作为回退
  3. 如果两种方法都失败，返回文字 `"PARSE_ERROR"` 而不是静默返回垃圾

**额外**：在 `get_wns` 中、在 `puts {get_wns_start}` 刷新之前添加一个 `sync_after_timeout(proc)` 调用，以在运行 Tcl 命令之前耗尽任何挂起的缓冲区。

### 变更 3：限制 get_completion 中的内部工具循环

**文件**：`dcp_optimizer.py`

在类级别添加常量：`MAX_TOOL_ROUNDS_PER_ITERATION = 15`

在 `get_completion()` (行 1774) 中，添加计数器并在超出限制时断开连接：
```python
tool_round = 0
while True:
    tool_round += 1
    if tool_round > self.MAX_TOOL_ROUNDS_PER_ITERATION:
        logger.warning(f"工具轮次 ({tool_round}) 超过限制，强制退出")
        content = "达到工具轮次上限"
        is_done = False
        break
```

### 变更 4：从 get_completion 中移除冗余的 global_no_improvement 递增

**文件**：`dcp_optimizer.py` (行 1911-1921)

移除 `self.global_no_improvement += 1` (行 1921)。`get_completion()` 中的 WNS 检查应仅更新 `failed_strategies` 和日志。计数器由 `_on_iteration_end()` 独占管理，使用权威的迭代前后 `best_wns` 比较。

### 变更 5：在 MemoryManager.add_tool_result 中添加防御性 WNS 检查

**文件**：`context_manager/manager.py` (行 270-295)

在 `add_tool_result()` 中、在 `wns > self._best_wns` 比较之前添加合理性检查。作为防御层，以防 WNS 未在 `dcp_optimizer.py:call_tool()` 中被捕获。

添加 `_wns_is_plausible(wns: float) -> bool`：
```python
def _wns_is_plausible(self, wns: float) -> bool:
    _best = self._best_wns if self._best_wns > float('-inf') else None
    if _best is None:
        return True
    if abs(wns) > max(abs(_best) * 10 + 10, 100):
        return False
    return True
```

### 变更 6：速率限制时的更智能模型切换

**文件**：`dcp_optimizer.py` (行 2134-2140)

在 429 速率限制异常处理程序中，在切换回迭代顶部之前显式设置模型为 `self.model_planner`。这避免了 `_select_model()` 可能再次选择速率限制的 Flash 模型的场景。

```python
except openai.RateLimitError:
    logger.warning("Flash 模型速率限制，强制切换到 Pro")
    current_model = self.model_planner
    # 重新尝试当前迭代而不是失败
    continue
```
